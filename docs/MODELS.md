# Models

The catalogue of compiled BPU models (`.hbm`) the examples, tests and benchmark
suite consume. This file is the single place that records, per model, **what it
is, where it came from, which build to take, and under what licence** — code
and the other docs link here rather than restating it.

It deliberately carries **no measurements**: timings, model sizes and memory
footprints live in [`benchmarks/RESULTS.md`](../benchmarks/RESULTS.md), which is
generated by `scripts/board_bench.py` on a clean board. Two hand-maintained
copies of the same numbers is how they drift.

The code references every one of these **only** by the repo-relative path
`models/<name>.hbm`, overridable per task through the environment
(`BCDL_MODELS`, `BCDL_OCR_DET`, …). No absolute path appears in the library,
the examples or the tests.

**OCR defaults to PP-OCRv6**; the v5 rows are kept as a fallback and for the
angle classifier, which v6 does not ship. The v6 recogniser is the **all-int16**
build on purpose: on the S100P it roughly halves the character error of the
compiler's default mixed-precision int8 build *and* is ~2× faster, because a
uniform int16 graph has no int8↔int16 requant on its boundaries. For text lines
longer than ~6.7:1 there is also a 960-wide v6 rec build
(`ppocrv6_medium_rec_int16_48x960.hbm`, ~2.4× the latency); point `BCDL_OCR_REC`
at it. The recogniser reads its class count from the model, so the 18385→18710
dictionary change needed no code — only `data/ppocr_keys_v6_18710.txt`. Recipes
for every v6 build live in the companion `bcdl-model-zoo` repo.

## Which build to take

**A model's file name matches the model name inside it, deliberately.** Several
entries above have sibling builds that differ only in how they were calibrated
or where the graph was cut, and the wrong one does not fail loudly — it compiles
cleanly, loads, and decodes to garbage or to quietly degraded accuracy. The
suffix is which build you have, not noise:

| suffix | model | what the other build does |
|---|---|---|
| `_aligned` | ArcFace | Calibrated on **aligned** face crops. The same-shaped build calibrated on centre crops is a different file, and the published accuracy figures are the aligned one's. |
| `_cut` | YOLOP | The published export bakes the anchor decode into the graph via `ScatterND`. That compiles without a single error into a model whose objectness and class columns are **never written**. Cutting the graph before the decode and emitting the three raw heads is what works. |
| `_v3` | PIDNet-S | The earlier builds were calibrated on data that had **not** been pre-normalised. When the calibration data is float32, the compiler's `norm_type` does not apply to it, so the input thresholds come out wrong — again with a clean compile. |
| `_qat` | OSNet-AIN | int8 PTQ does not work on this network at all: it compiles cleanly and returns well-formed unit vectors whose Market-1501 Rank-1 is 51% against the float model's 85%. The shipped build was recovered by QAT self-distillation to 84.6%. |
| `_crop` / `_int16` | Lite Any Stereo V2 | Two calibration modes — centre-crop-fit and resize-fit. Match it to how you letterbox the pair; the repo's stereo test uses a centre-cropped pair and prefers `_crop`. |

Two more choices in the table are **not** better/worse pairs:

- **Real-ESRGAN Compact vs SPAN** — Compact is perceptual and wins on blurred or
  compressed input; SPAN is fidelity-oriented, wins on clean input, and is a
  sixth of the size. Keep both.
- **The 128×128 super-res tile is deliberate.** A compiled `.hbm` is mostly
  instruction stream, not weights, and it scales with tile **area** — the same
  network at 256×256 is a 148 MB model against 37 MB here, for identical
  per-pixel throughput. If the runtime already tiles, use the small tile.

To verify a fetch actually gave you these builds: empty `models/`, re-run the
fetch script, and run the test suite.

## Getting the models

The `.hbm` files are **not committed** (too large; `.gitignore` excludes
`*.hbm`). Populate `models/` on the board with:

```bash
scripts/fetch_models.sh
```

That script is the single place the local source paths live — set them in
`scripts/local.env` (gitignored; see `scripts/local.env.example`). Anything
unset falls back to a placeholder and is simply reported as `MISSING`.

Model conversion (ONNX → `.hbm`, PTQ calibration) happens **offline on an x86
host** with the D-Robotics OpenExplorer toolchain. The full conversion recipes —
ONNX export, calibration, `hb_compile` config and the accuracy/latency numbers
for every model above — live in the companion
[**bcdl-model-zoo**](https://github.com/ruisv/bcdl-model-zoo) repository.

Each `.hbm` is compiled for a specific BPU **march**, so run the build compiled
for your board — S100 and S100P share the Nash march, S600 is compiled for its
own. The BCDL runtime itself is identical across the S-series.

## Licensing

The models above are third-party and carry **their own licences**. BCDL's own
code (this repository, Apache-2.0) is unrelated to them: it is a general-purpose
runtime that loads any `.hbm`. The licences below constrain **the model weights
and their compiled `.hbm`**, so check each one before redistributing or shipping
commercially.

| upstream | licence |
|---|---|
| PP-OCRv6 det/rec, PP-OCRv5 det/rec, PP-LCNet cls (PaddleOCR) | Apache-2.0 |
